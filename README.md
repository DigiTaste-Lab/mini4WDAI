# Mini4AI MG24 Rules Realtime — v3.7

XIAO MG24 Sense + LSM6DS3 IMU を使った、ジャイロベース・パターンマッチング型のミニ四駆 AI 制御システム。BLE 経由で Web UI からルールを書き込み、走行中はファーム単独で **セグメント検出 → ルール照合 → アクション発火** を行う。LDA 等の機械学習推論は使わず、明示的なルールベースで動作する。

## 構成

| ファイル | 役割 |
|---|---|
| `Mini4AI_MG24_Rules_Realtime.ino` | XIAO MG24 ファーム (Arduino) |
| `index.html` | 単一ファイルの Web UI (Web Bluetooth) |

ボード: Seeed XIAO MG24 Sense
IMU: LSM6DS3 (XIAO Sense 内蔵)
通信: BLE GATT (`12345678-1234-5678-1234-56789abcdef0` 配下)
制御周期: 200 Hz (5 ms)

## セグメントイベント

ジャイロ z 軸 (gz) の符号と振幅から走行区間を **3族 × 各3イベント = 9種** に分類してリアルタイム発火する。

### カーブ系 (左C類 / 右C類)

カーブには「入口」「中盤」「下降中」「出口」の4フェーズがある。出口を除く3フェーズで **イベントが1回ずつ発火** する。

| フェーズ | 90° カーブ | 180° カーブ | 発火条件 |
|---|---|---|---|
| 入口 | 左C / 右C | 左C / 右C | `\|gz\| > CORNER_TH` を超えた瞬間 |
| 中盤 | (発火なし) | 左C継続 / 右C継続 | カーブ突入から `t = CURVE_LONG_MS` 経過 |
| 下降中 | 左ピーク後 / 右ピーク後 | 左ピーク後 / 右ピーク後 | ピーク値から `CURVE_PEAK_DROP_DPS` 下降した瞬間 |
| 出口 | (発火なし) | (発火なし) | `\|gz\| < STRAIGHT_TH` が `EXIT_DEBOUNCE_MS` 継続 |

ポイント:

- **「○ピーク後」は出口ではなく出口の手前**。ピーク値から一定量下降したタイミングで発火するので、まだカーブ内にいる
- **「○C継続」は90°では発火しない**。`CURVE_LONG_MS` (既定 180ms) に届く前にカーブが終わるため
- 出口判定そのものは独立イベントを発火しない (デバウンスのみ)

ブレーキ用途の典型パターン:

- 「カーブで減速したい」 → `左C / 右C` (入口で即発火)
- 「長カーブだけブレーキを継続したい」 → `左C継続 / 右C継続` (180° のみ反応)
- 「カーブ抜けの直前にブレーキを切りたい」 → `左ピーク後 / 右ピーク後`

### 直線系 (直線類)

| イベント | 発火条件 |
|---|---|
| 直線継続 | 直線が `STRAIGHT_HOLD_MS` 以上続いた瞬間に **1回だけ** 発火 (再発火しない) |
| ウェーブ | 左右スパイクが `WAVE_WINDOW_MS` 以内に連続検出された瞬間に発火 (`WAVE_COOLDOWN_MS` のクールダウンあり) |

直線そのもの (`SEG_STRAIGHT = 1`) は背景状態として保持されるが、独立イベントとしては発火しない。

## ルールエンジン

各ルールは **パターン (1〜8セグメント) + 発火条件 + アクション列** で構成される。最大8本まで保持可能。

### ルール構造

| 項目 | 値 |
|---|---|
| パターン | セグメントの順序列 (例: `[左C, 直線継続, 右C]`) |
| トリガモード | 毎回 / 奇数回 / 偶数回 / N回毎 / N回目のみ |
| 優先度 | 1〜255、大きいほど先にマッチ判定 |
| アクション列 | ブレーキ / 速度調整 / 停止 / 待機 を最大5本連結 |
| ループモード | ON で繰り返しマッチを許可 |
| Per-rule タイミング上書き | `peakAfterMs` / `straightHoldMs` / `curveLongMs` をルール単位で上書き可能 (空欄=共通値継承) |

### アクション種別

| 種別 | 動作 |
|---|---|
| ブレーキ | 指定 ms の間、PWM Duty を 0 (もしくは指定 strength) に下げる |
| 速度調整 | 指定 strength で速度を変更 |
| 停止 | 完全停止 |
| 待機 | 何もせず指定 ms 経過してから次のアクションへ |
| アクション無効 | 指定 ms の間、**他ルール含めて全アクションをグローバル抑制**。直前のモータ出力を維持したまま走行継続。スタート直後の不安定区間や、ジャンプ着地後に「ブレーキかけたくない」場面で使用 |

### 優先度とプリエンプション

各ルールは `priority` (1〜255) を持ち、走行中に複数ルールがマッチした場合の調停に使われる。動作は **「発動順」ではなく「プリエンプション (横取り)」** で、以下の通り:

| 状況 | 結果 |
|---|---|
| 何も走ってない | 新ルールが発動 |
| 走行中ルール (prio A) より新ルール (prio B) が **高い** (B > A) | 新ルールが横取り。Aは中断 |
| 新ルールのほうが **低い** (B < A) | 新ルールは無視される (Aが完走) |
| 同じ優先度 (B == A) | リスト上で先に並んでるほうが勝つ (後発は無視) |

実用的な設計指針:

- **絶対に効かせたい安全系のブレーキ**: priority 高め (例: 200)
- **通常のセクション制御**: priority 普通 (例: 100)
- **フォールバック的な汎用ルール**: priority 低め (例: 50)

### LOCK ブロックの仕組み

`アクション無効` (LOCK) は優先度ロジックの **外側** で動作する:

- LOCK 発動中は、**他ルールの優先度に関係なく**、新規アクション (ブレーキ/速度/停止) の発火を全部ブロックする
- セグメント検出、ルールマッチ、イベントログは継続する (ログには見える)
- モータ出力は LOCK 直前の状態を維持
- LOCK 自身は WAIT 同様に時間消化型のアクション。LOCK の後ろに他のアクションを連結しても、LOCK 中は他ルールに横取りされない
- **例外**: 先頭アクションが LOCK のルールは LOCK 中でも発火可 (再アーム/延長用途)

つまり LOCK は「最強の上書き」で、優先度ピラミッドより上に位置する。緊急停止が必要なシーンには LOCK を使わない別ルールを使うこと。

## Web UI 使い方

3つのタブ:

1. **走行**: マシン接続、開始/停止、リアルタイムイベントタイムライン、停止後ログ
2. **設計**: ルール編集、QR / JSON 共有
3. **認識調整**: 全パラメータ (CORNER_TH, STRAIGHT_TH, WAVE 系, CURVE_LONG_MS など)

### 設計タブ — 解説モーダル

各ルールのセグメントパレットは **左C類 / 右C類 / 直線類** の3グループに分かれていて、各グループのヘッダに **💡 タイミング** ボタンがある。クリックすると、そのグループのイベントが時間軸上のどこで発火するかを SVG 図示する解説モーダルが開く。

解説モーダル内では3族のタブを切り替えて閲覧することもできる。

### 設定共有

設定 (パラメータ + ルール) は3経路で共有可能:

- **JSON ファイル**: エクスポート / インポート
- **QR**: 表示 (生成側) / 読込 (カメラ・画像・テキスト貼付の3方式)
- **localStorage**: 自動保存、起動時に復元

## デフォルトパラメータ

| パラメータ | 既定値 | 単位 | 役割 |
|---|---|---|---|
| `CORNER_TH` | 260.0 | dps | カーブ入口判定の閾値 |
| `STRAIGHT_TH` | 220.0 | dps | 直線復帰判定の閾値 |
| `EXIT_DEBOUNCE_MS` | 15 | ms | 出口デバウンス |
| `CURVE_PEAK_AFTER_MS` | 30 | ms | ピーク通過後の最小経過時間 |
| `CURVE_PEAK_DROP_DPS` | 40.0 | dps | ピーク値からの最小下降量 |
| `CURVE_LONG_MS` | 180 | ms | 長カーブ判定 (中盤イベント) の閾値 |
| `STRAIGHT_HOLD_MS` | 250 | ms | 直線継続判定 |
| `WAVE_POS_TH` / `WAVE_NEG_TH` | 200.0 | dps | ウェーブのスパイク閾値 |
| `WAVE_WINDOW_MS` | 200 | ms | ウェーブ検出窓 |
| `WAVE_LATCH_MS` | 1200 | ms | ウェーブラッチ時間 |
| `WAVE_COOLDOWN_MS` | 400 | ms | ウェーブ再発火クールダウン |

## v3.7 変更履歴

### アクション無効 (ACTION_LOCK) を追加

新しいアクション種別 `アクション無効` (内部値 6) を追加。指定 ms の間、**他ルールを含めた全アクション** (ブレーキ / 速度変更 / 停止) のグローバル発火を抑制する。

- セグメント検出・ルールマッチ・イベントログは継続
- モータ出力は LOCK 直前の状態を維持(`NORMAL_DUTY` への巻き戻しもしない)
- LOCK 中でも、**先頭アクションが LOCK のルール** は再ロックのため発火を許可(再アーム可能)
- 走行開始 / 停止時に LOCK 状態は自動クリア

典型用途: スタート直後の不安定区間、ジャンプ着地後の誤ブレーキ防止、特定セクションの「素通し」化。

## v3.6 変更履歴

### LONG_PEAK セグメント廃止

旧 `SEG_LEFT_LONG_PEAK` (12) / `SEG_RIGHT_LONG_PEAK` (13) を削除。理由:

- LONG_PEAK は PEAK と発火タイミングがほぼ同じ (出口手前の下降中)
- 「長カーブ判別」は HOLD が既に担当しているため、PEAK と HOLD の組み合わせで完全にカバーできる
- 教材としても「入口/中盤/下降中/出口」の3フェーズ構成のほうが学生に伝わりやすい

これにより、LONG_PEAK 関連の状態変数 (`longPeakEmitted`)、発火ブロック、override 集約コードを全て削除。バージョンも v3.5 → v3.6 に。

### 既存ルールの自動マイグレーション

パターンに 12 / 13 を含む既存ルールは、ロード時 (localStorage / JSON / QR) に自動で 7 (LEFT_PEAK) / 8 (RIGHT_PEAK) に降格。`migrateRules()` 関数で処理。

### UI 改善

- **族別パレット**: ルール編集時のセグメントタイルを左C類 / 右C類 / 直線類の3グループにグルーピング表示
- **解説モーダル**: 各族グループのヘッダに 💡 タイミングボタンを設置。SVG 図示でフェーズ (入口/中盤/下降中/出口) を可視化
- **凡例の整理**: 認識調整タブの凡例から長カーブ系 pip を削除

### バグ修正

- **QR モーダルの初期化エラー**: モーダル HTML をスクリプトより前に配置することで `Cannot set properties of null` を解消
- **QR ライブラリの差し替え**: cozmo/node-qrcode (CDN ビルド不安定) → neocotic/qrious (純JS canvas、安定動作)
- **QR スキャナの精度向上**: jsQR の `inversionAttempts` を `'attemptBoth'` に変更、`willReadFrequently:true` で getImageData の警告解消
